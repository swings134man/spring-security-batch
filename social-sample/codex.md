# codex.md

## 작업 범위

- 모든 작업은 현재 샘플 프로젝트 디렉토리 내부에서만 수행한다.
- 본 모듈은 `Spring Boot + Kotlin` 기반의 **소셜 로그인 기본 샘플 프로젝트**를 만드는 것이 목적이다.
- 이 프로젝트는 추후 다른 프로젝트에 그대로 복사·확장할 수 있는 **기본 템플릿** 역할을 해야 한다.
- 작업 대상은 백엔드, 서버 렌더링 프론트(Thymeleaf), 인증 흐름, Redis 연동, 예외 처리, 문서 정리까지 포함한다.

## 프로젝트 정의

- 프로젝트명: `social-sample`
- 목적: 국내 서비스에서 일반적으로 사용하는 방식의 **OAuth 2.0 + JWT + Redis 기반 로그인 샘플** 구현
- 핵심 시나리오:
  - 일반 사이트 로그인(아이디/비밀번호) 지원
  - 카카오 소셜 로그인 지원
  - 네이버 소셜 로그인 지원
  - 소셜 로그인 최초 진입 시 추가 회원정보 입력 후 가입 완료 처리
  - 자체 JWT Access Token / Refresh Token 발급 및 검증
  - Refresh Token은 Redis에만 저장
  - Thymeleaf 기반 로그인/추가정보 입력/테스트 화면 제공
  - 재사용 가능한 구조와 패키지 설계를 우선한다

## 기술 스택

- 언어: Kotlin
- 런타임: Java 17 이상
- 프레임워크: Spring Boot 3.x
- 템플릿 엔진: Thymeleaf
- 보안: Spring Security
- 로그인 프로토콜: OAuth 2.0 Authorization Code 방식
- 토큰: JWT
- 저장소:
  - RDBMS 사용 전제(로컬: h2, 실제서비스: postgreSQL)
  - Redis 사용 (Refresh Token 저장)
- 빌드: Gradle Kotlin DSL

## 구현 목표

Codex는 아래 목표를 만족하도록 모듈을 구성한다.

1. 소셜 로그인과 일반 로그인을 한 프로젝트 안에서 함께 지원한다.
2. 인증과 회원 도메인 정보를 분리된 책임으로 설계한다.
3. 소셜 로그인은 외부 인증을 사용하되, 최종 서비스 인증은 우리 JWT로 수행한다.
4. 최초 소셜 로그인 사용자는 추가 회원정보 입력이 완료되어야 정상 가입 상태가 된다.
5. Access Token은 1시간, Refresh Token은 7일 기준으로 동작한다.
6. Refresh Token은 Redis에 저장하고, Access Token은 저장하지 않는다.(사용자에게만 추후 Bearer token 형태로 전달받고 인증할것)
7. 샘플 프로젝트지만 실제 서비스에 가까운 구조와 흐름을 따른다.
8. 다른 프로젝트에 쉽게 복사 가능한 수준으로 모듈화한다.(설정, 구조, 환경에 따른 분리)

## 작업 정의

Codex 작업은 아래 순서로 진행한다.

1. 프로젝트의 기본 구조를 설계한다.(Gradle 설정, 기본 패키지 구조, 공통 설정, 추가적으로 필요한 테이블 정의(schema.sql))
2. 패키지 구조를 인증, 회원, 공통, 웹 계층으로 나눈다.(이미 생성된 패키지 구조를 참조함.)
3. Spring Security 기본 설정을 구성한다.(프로파일 별로 설정분리 가능하도록{h2 console 허용 등})
4. 일반 로그인 흐름을 구현한다.(회원가입 부터 로그인, JWT 발급까지)
5. 카카오/네이버 OAuth 2.0 로그인 흐름을 구현한다.(환경변수로 API Key 관리예정)
6. 소셜 로그인 후 사용자 존재 여부 및 가입 상태를 판별하는 로직을 구현한다.
7. 추가 회원정보 입력 흐름을 구현한다.
8. JWT 발급/검증/재발급/로그아웃 흐름을 구현한다.
9. Redis 기반 Refresh Token 저장 및 검증 로직을 구현한다.
10. Thymeleaf 화면을 통해 로그인/추가정보 입력/성공 확인이 가능하도록 한다.
11. 예외 처리, 공통 응답, 테스트 가능한 구조를 정리한다.
12. 실행 및 검증이 가능한 샘플 상태로 마무리한다.

## 패키지 구조 가이드

패키지 구조는 기능 중심으로 나눈다. 
필요 시 하위 패키지로 세분화할 수 있지만, 과도한 분할은 피한다.
- models: 안에 도메인 패키지 그안에 entity, repository, service 패키지로 나누어 구현한다
- rest: 컨트롤러와 DTO 패키지로 나눈다
- infra
  - common: 공통 유틸, 예외, 응답 등
  - security: 시큐리티 관련된 내용들(설정, JWT 등)
  - configs: 레디스 및 기타 설정 관련 클래스
  - utils: 유틸 클래스

## 작업 시 준수 사항

- OOP 원칙을 고려한다.
- 클래스 책임을 명확히 분리한다.
- Spring Security 설정은 역할별로 읽기 쉽게 구성한다.
- OAuth 관련 코드는 provider 확장이 가능하도록 작성한다.
- 카카오/네이버 provider 로직은 공통화 가능한 부분과 개별 구현 부분을 구분한다.
- JWT/Redis/회원가입 완료 상태 로직이 서로 섞이지 않게 분리한다.
- Thymeleaf는 샘플 UI 수준으로만 구현하되 흐름 검증이 가능해야 한다.
- 비밀번호 저장 시 평문 저장은 금지한다.(BCrypt 해시 사용)
- 민감한 값(client secret, jwt secret 등)은 환경변수 또는 설정 분리 전제를 따른다.
- 실제 운영 확장이 가능한 구조를 우선한다.

## Codex가 특히 신경써야 할 것

- 메인페이지는 모두가 진입가능 그외에는 모두 로그인 필요한 구조로 만들것.
  - 페이지는 샘플로 만들것.
- 일반 로그인과 소셜 로그인을 한 인증 체계 안에서 자연스럽게 공존시키기
- 소셜 로그인 사용자의 최초 가입 흐름과 재로그인 흐름을 명확히 분리하기
- JWT는 서비스 인증 토큰이고, 소셜 access token은 외부 사용자 정보 조회용이라는 점을 코드 구조에 반영하기
  - 소셜 Access Token 은 인증 흐름에서 provider 사용자 정보 조회에만 사용되고, JWT는 최종 서비스 인증에만 사용되도록 책임 분리하기
- Refresh Token은 Redis에만 저장하고, Access Token은 서버에 저장하지 않는 구조 유지하기
- 추후 복붙 재사용이 가능한 수준으로 설정/구조/네이밍을 정리하기
