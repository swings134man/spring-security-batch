# explain.md

## 문서 목적

이 문서는 `Spring Boot + Kotlin` 기반의 소셜 로그인 샘플 모듈에 들어가야 할 기능과 정책을 정리한다.
이 프로젝트는 단순 데모가 아니라, 향후 여러 프로젝트에 복사해 사용할 수 있는 **기본 인증 모듈 템플릿**을 목표로 한다.

---

## 1. 전체 목표

이 샘플 모듈은 아래를 동시에 제공해야 한다.

- 일반 사이트 로그인
- 카카오 로그인
- 네이버 로그인
- 자체 JWT 기반 인증
- Redis 기반 Refresh Token 관리
- 최초 소셜 로그인 사용자의 추가 정보 입력 흐름
- 실제 서비스에 가까운 회원 상태 관리

즉, 이 샘플은 단순히 "소셜 로그인 버튼이 동작하는 프로젝트"가 아니라,
**회원가입, 로그인, 재로그인, 추가 정보 입력, 토큰 발급, 토큰 재발급, 로그아웃**까지 연결되는 기본 인증 모듈이어야 한다.

---

## 2. 왜 이런 구조가 필요한가

소셜 로그인에서 제공하는 정보만으로는 실제 서비스 운영에 필요한 정보를 모두 충족할 수 없다.
예를 들어 소셜 서비스는 이메일, 닉네임, provider 식별자 정도만 줄 수 있지만,
우리 서비스는 다음과 같은 별도 정보가 필요할 수 있다.

- 서비스 닉네임
- 전화번호
- 약관 동의 여부
- 마케팅 수신 동의 여부
- 프로필 이미지
- 상태값(가입완료 여부, 차단 여부 등)
- 서비스 전용 설정 정보

따라서 인증 정보와 서비스 회원 정보를 분리해서 생각해야 한다.

---

## 3. 설계 원칙

### 3.1 인증과 회원정보 분리
소셜 로그인은 외부 서비스가 인증을 담당한다.
하지만 서비스 도메인에 필요한 회원 정보는 우리 서비스가 직접 관리해야 한다.

즉 개념적으로는 아래처럼 나뉜다.

- 외부 인증 정보
- 우리 서비스 회원 정보

### 3.2 소셜 access token은 우리 인증 토큰이 아니다
카카오/네이버에서 발급한 access token은
사용자 정보를 조회하거나 외부 API를 호출하기 위한 토큰일 뿐이다.

우리 서비스의 인증은 최종적으로 **우리 JWT**로 처리해야 한다.

### 3.3 최종 로그인 상태는 우리 서비스가 관리한다
소셜 로그인 성공만으로 곧바로 서비스 가입 완료가 되는 것이 아니라,
우리 서비스 기준으로 필요한 정보가 다 채워졌는지 확인한 뒤 최종 가입 완료 상태를 관리해야 한다.

---

## 4. 로그인 방식

이 샘플은 아래 두 가지 로그인 방식을 동시에 지원해야 한다.

### 4.1 일반 로그인
- 이메일 기반
- 비밀번호 기반 인증(BCrypt 해시 저장)
- Spring Security 인증 절차 사용
- 성공 시 JWT 발급

### 4.2 소셜 로그인
- 카카오 OAuth 2.0
- 네이버 OAuth 2.0
- Authorization Code 방식 사용
- 서버가 인가코드를 받아 access token 교환
- access token으로 사용자 정보 조회
- 이후 우리 서비스 사용자와 매핑
- 성공 시 JWT 발급
- 우리쪽 DB 에 정보없을 경우 추가 정보 입력 페이지 이동 -> 입력 완료 후 DB 저장 및 JWT 발급

---

## 5. OAuth 2.0 흐름에서 반드시 포함해야 할 개념

샘플 모듈은 아래 개념이 반영되어야 한다.

### 5.1 인가코드 기반 처리
소셜 로그인 후 프론트는 인가코드를 전달받고,
(내 애플리케이션)서버는 그 인가코드를 가지고 provider에 access token을 요청한다.

### 5.2 사용자 정보 조회
provider에서 받은 access token으로 사용자 정보를 조회한다.

### 5.3 provider 사용자 식별자 기준 처리
사용자 식별은 이메일보다 provider의 고유 사용자 식별자를 중심으로 처리해야 한다.
이메일은 변경되거나 제공되지 않을 수 있기 때문이다.

### 5.4 최종 서비스 JWT 발급
provider access token으로 로그인 상태를 유지하지 않고,
우리 서비스의 JWT Access Token / Refresh Token 체계로 전환해야 한다.

---

## 6. JWT 정책

### 6.1 Access Token
- 유효기간: 1시간
- 서버 저장 없음
- 요청 인증용으로 사용
- 회원 이메일, 닉네임, role 등 최소한의 정보만을 포함해야 한다. (보안상 민감한 정보는 포함하지 않는다)

### 6.2 Refresh Token
- 유효기간: 7일
- Redis에 저장
- 재발급 및 로그아웃 처리에 사용
- AT 만료시 RT 검증 후 새로운 AT 발급이 가능해야 함

### 6.3 토큰 정책이 분리되어야 하는 이유
- Access Token은 짧게 유지하고 빠르게 검증해야 한다.
- Refresh Token은 서버에서 추적 가능해야 한다.
- 로그아웃, 재로그인, 강제 만료 시 Redis 기반 관리가 필요하다.

---

## 7. Redis 역할

이 샘플에서 Redis는 인증 관점에서 아래 역할을 가진다.

- Refresh Token 저장
- Refresh Token 검증
- 로그아웃 시 Refresh Token 삭제
- 재발급 시 기존 Refresh Token 교체

주의할 점:
- Access Token은 Redis에 저장하지 않는다.
- Redis TTL과 JWT 만료 시간 단위를 헷갈리지 않도록 분리해서 설계한다.

---

## 8. 회원 상태 관리

소셜 로그인은 성공했지만, 서비스 가입이 아직 완료되지 않은 상태를 표현할 수 있어야 한다.

예를 들면 아래 같은 상태를 둔다.

- `PENDING`: 소셜 인증은 완료됐지만 추가 정보 입력 전
- `ACTIVE`: 가입 완료
- `BLOCKED`: 차단
- `WITHDRAWN`: 탈퇴

샘플 모듈에는 최소한
**최초 소셜 로그인 후 PENDING → 추가 정보 입력 후 ACTIVE 전환** 흐름이 있어야 한다.

---

## 9. 최초 소셜 로그인 사용자 처리

최초 소셜 로그인 사용자는 아래 흐름을 가져야 한다.

1. 카카오/네이버 로그인 성공
2. provider 사용자 정보 조회
3. 우리 서비스에 매핑된 계정이 있는지 확인
4. 없으면 임시 사용자 상태 또는 가입 대기 상태로 생성
5. 추가 회원정보 입력 페이지로 이동
6. 필수 정보 입력 완료
7. 가입 완료 처리
8. 최종 로그인 완료

이 흐름이 반드시 존재해야 샘플로서 가치가 있다.

---

## 10. 추가 회원정보 입력에서 들어가야 할 개념

테이블 구조는 여기서 정의하지 않지만,
샘플 프로젝트에는 아래 개념이 반영되어야 한다.

- 소셜 로그인으로는 확보되지 않는 서비스 전용 정보
- 필수 입력값과 선택 입력값 구분
- 약관/개인정보/마케팅 동의 여부 관리 가능성
- 가입 완료 전까지 제한된 상태 처리

샘플인 만큼 모든 실서비스 필드를 다 구현할 필요는 없지만,
**추가 정보 입력이 실제 서비스에서 왜 필요한지**가 드러나야 한다.

---

## 11. 회원 도메인에서 샘플에 포함되면 좋은 개념

이 샘플 모듈은 이후 프로젝트에 복사될 예정이므로,
단순 이메일/닉네임 수준보다 조금 더 현실적인 사용자 모델을 염두에 둬야 한다.

샘플 차원에서 고려할 만한 정보:

- 이메일
- 비밀번호(일반 로그인용)
- 닉네임
- 휴대폰 번호
- 프로필 이미지 URL
- 가입 상태
- provider 종류
- provider 사용자 식별자
- 마지막 로그인 시각
- 생성일/수정일

---

## 12. 일반 로그인과 소셜 로그인의 공존

샘플 모듈은 아래 상황을 자연스럽게 처리할 수 있어야 한다.

- 일반 회원가입 사용자 로그인
- 소셜 로그인 사용자 로그인
- 같은 서비스 회원이 여러 로그인 수단을 가질 수 있는 확장 가능성

초기 샘플은 카카오/네이버 + 일반 로그인만 구현하더라도,
구조 자체는 이후 구글, 애플, 추가 provider를 넣기 쉽게 설계되어야 한다.

---

## 13. 화면 구성에 포함되어야 할 최소 페이지

Thymeleaf 기반으로 아래 화면은 최소 포함되는 것이 좋다.

- 메인 또는 홈 화면
- 일반 로그인 화면
- 회원가입 화면(일반 로그인용)
- 소셜 로그인 진입 화면
- 추가 정보 입력 화면
- 로그인 성공 후 확인 화면
- 토큰 재발급 또는 인증 확인 테스트용 화면

디자인은 단순해도 되지만 흐름 검증이 가능해야 한다.

---

## 14. 예외 처리에서 샘플에 반드시 있어야 할 것

샘플이지만 아래 예외 케이스는 최소한 처리되어야 한다.

- provider access token 교환 실패
- 사용자 정보 조회 실패
- 잘못된 authorization code
- 만료된 Refresh Token
- Redis에 토큰이 없는 경우
- 가입 대기 상태 사용자 접근 제한
- 차단된 사용자 접근 제한

이 예외들이 없으면 샘플을 복사해갈 때 실제 프로젝트에서 바로 문제가 생긴다.

---

## 15. 보안 측면에서 포함되면 좋은 것

- 비밀번호는 해시 저장
- JWT secret 외부 설정 분리
- OAuth client secret 외부 설정 분리
- 인증이 필요 없는 경로와 필요한 경로 구분
- 로그인 완료 전 임시 상태 접근 제한
- 로그아웃 시 Refresh Token 제거
- 재발급 시 토큰 일치 여부 검증

샘플이어도 이 정도 기본선은 들어가는 것이 좋다.

---

## 16. 로그와 디버깅 포인트

샘플 프로젝트는 학습/재사용 목적이 크므로,
최소한 아래 로그 포인트는 있어야 한다.

- 일반 로그인 성공/실패
- 카카오/네이버 로그인 시작/성공/실패
- provider 사용자 정보 조회 성공/실패
- 최초 로그인 여부 판별 결과
- 추가 정보 입력 완료 여부
- JWT 발급 / 재발급 / 로그아웃 처리
- Redis Refresh Token 저장/삭제

다만 access token, refresh token 원문 전체를 로그로 남기면 안 된다.

---

## 17. 샘플 프로젝트에 있으면 좋은 추가 요소

아래는 꼭 필수는 아니지만, 샘플 완성도를 높여준다.

- 통합된 인증 성공 응답 구조
- 공통 예외 응답 구조
- provider별 사용자 정보 매핑 DTO 분리
- 테스트용 로그인 확인 API
- 현재 로그인 사용자 정보 확인 API
- 재발급 테스트 API
- 로그아웃 API

이런 요소가 있으면 추후 복붙해서 확장하기 쉬워진다.

---

## 18. 이 샘플이 지향해야 하는 수준

이 샘플은 단순 예제보다 한 단계 위 수준이어야 한다.

즉,

- 블로그 튜토리얼 수준의 최소 구현이 아니라
- 실제 회사 프로젝트에 들어갈 수 있는 구조를 흉내 내는 수준
- 다만 과도하게 복잡하지는 않게
- 그대로 가져다가 빠르게 확장 가능한 형태

를 목표로 한다.

---

## 19. 이번 문서에서 일부러 제외한 것

아래 항목은 이번 문서에서 다루지 않는다.

- SQL 작성
- 마이그레이션 스크립트
- 인프라 배포 구성
- 운영환경 보안 고도화

이번 문서는 어디까지나
**샘플 인증 모듈에 어떤 기능과 개념이 들어가야 하는지**를 정리하는 문서다.

다만, DB 관련 스크립트 및 SQL 은 별도의 파일에서 정의한다 (예: `schema.sql`, `data.sql`, `migration/` 폴더 등)로 분리해서 관리할것이다.

---

## 20. 최종 요약

이 샘플 모듈에는 아래가 반드시 들어가야 한다.

- Spring Security 기반 인증 구조
- 일반 로그인
- 카카오 로그인(API 키는 환경변수로 분리)
- 네이버 로그인(API 키는 환경변수로 분리)
- OAuth 2.0 Authorization Code 흐름
- 소셜 로그인 성공 후 사용자 조회/생성/상태 분기
- 추가 회원정보 입력 흐름
- JWT Access Token (1시간)
- Refresh Token (7일, Redis 저장)
- 로그아웃/재발급 흐름
- Thymeleaf 기반 검증 가능한 화면
- 실제 서비스에 가까운 회원 상태 관리

이 기준을 만족하면,
이후 다른 프로젝트에 인증 기본 모듈로 복사해 사용할 수 있다.
